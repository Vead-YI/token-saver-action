# 节省效果基准

本文档给出当前实现下的代表性场景和预期节省区间，作为 README 中数据的落地说明。

## 1. Prompt 压缩

测试输入：
- 中文请求，包含礼貌用语、重复意图、列表重复项
- Markdown 代码块混合说明文字

预期结果：
- `light`: 主要去掉礼貌用语和空白，通常节省 5-15%
- `moderate`: 去重意图和列表项，通常节省 15-30%
- `aggressive`: 对说明文字进一步缩短，通常节省 20-40%

说明：
- 代码块默认尽量保留原样，避免节省 token 的同时破坏可执行上下文
- `aggressive` 适合模型上下文很紧张的情况

## 2. 文件读取

典型场景：
- 500 行 Python 文件，包含多个类、方法和实现细节

预期结果：
- `signatures`: 常见节省 70-90%
- `summary`: 常见节省 60-85%
- `keywords`: 当关键词命中集中时，可节省 80% 以上
- `head_tail`: 适合配置文件和日志，节省取决于文件总长度

说明：
- Python 文件会尽量按源码顺序保留类、方法签名和文档字符串
- `diff` 模式使用 unified diff，更适合多轮代码修改场景

## 3. 对话历史裁剪

典型场景：
- 15-30 条多轮对话
- 最近消息必须保留
- 较早的错误信息和约束条件仍然重要

预期结果：
- 常见节省 40-70%
- 比纯“从后往前截断”更不容易丢掉关键错误和需求

说明：
- 当前版本优先保留最近消息
- 对早期消息会根据角色、报错、约束、问题等特征做粗粒度打分
- 如需进一步压缩，可先把早期消息替换为摘要占位符再裁剪

## 4. 输出控制

典型场景：
- 编程助手、问答助手、Agent 工具执行总结

预期结果：
- 只注入简洁指令时，输出常见减少 15-25%
- 叠加句数预算和“代码优先”规则时，输出还会进一步收缩

说明：
- 最适合需要高频调用、且不希望模型输出长篇解释的场景
- 如果任务天然需要长推理或长文案，不建议预算设得过紧
